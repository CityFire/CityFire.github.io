软件工程

发表于2023-11-06|更新于2026-03-16|工程、架构
|阅读量:

软件工程是计算机科学的重要分支，它是研究和应用如何以系统性的、规范化的、可定量的过程化方法去开发和维护软件，以及如何把经过时间考验而证明正确的管理技术和当前能够得到的最好的技术方法结合起来的学科。软件工程涉及到程序设计语言、数据库、软件开发工具、系统平台、标准、设计模式等多个方面。

软件工程的核心目标是提高软件质量、降低开发成本、缩短开发周期。通过系统化的方法和工具，软件工程帮助开发团队更高效地构建可靠、可维护、可扩展的软件系统。

软件过程模型

软件过程模型是软件开发过程的抽象表示，描述了软件从需求获取到维护的完整生命周期中各个阶段的组织和顺序。常见的软件过程模型包括：

  • 瀑布模型：传统的线性顺序模型
  • V模型：瀑布模型的变种，强调测试
  • 原型模型：快速构建原型以获取需求
  • 螺旋模型：结合迭代和风险评估
  • 构件组装模型：基于构件的软件开发
  • 快速应用开发（RAD）：强调快速迭代
  • 统一过程（UP）：用例驱动的迭代模型
  • 敏捷开发方法：强调响应变化

瀑布模型

瀑布模型是最经典、最传统的软件开发过程模型，它将软件开发过程划分为多个顺序执行的阶段，每个阶段必须完成后才能进入下一阶段。

瀑布模型的特点

  • 严格区分阶段：需求分析、设计、实现、测试、维护等阶段因果关系紧密相连
  • 顺序执行：各阶段按固定顺序执行，只有完成当前阶段才能进入下一阶段
  • 文档驱动：每个阶段都要求产生详细的文档作为阶段完成的标志

瀑布模型的适用场景

瀑布模型适合需求明确、不易变化的项目，如嵌入式系统、操作系统、大型企业应用等。

瀑布模型的缺点

  • 需求难确定：软件需求的完整性和正确性很难在项目初期确定
  • 反馈周期长：严格的串行化导致很长时间才能看到可运行的系统
  • 不灵活：要求每个阶段一次性完全解决该阶段工作，这不现实

原型模型

原型模型是一种快速构建软件原型以获取用户需求的开发模型，特别适合需求不明确的项目。

原型模型的两个阶段

  1. 原型开发阶段：快速构建一个可以演示的系统原型
  2. 目标软件开发阶段：基于确定的原型，开发完整的软件系统

原型模型的类型

  • 抛弃型原型：原型仅用于需求验证，开发完成后被抛弃
  • 演化型原型：原型逐步演化为最终产品

原型模型的优点

  • 快速获取真实需求
  • 减少需求理解偏差
  • 提高用户参与度
  • 降低项目风险

原型相关模型

原型相关模型展示了不同原型开发方法之间的关系和适用场景。

V模型

V模型是瀑布模型的重要变种，它将测试活动与开发过程紧密结合，体现了”质量内建”的理念。

V模型的特点

  • 左边是开发阶段，右边是测试阶段
  • 开发阶段与测试阶段一一对应
  • 强调早期测试和持续测试
  • 单元测试对应详细设计
  • 集成测试对应概要设计
  • 系统测试对应需求分析

V模型的优势

  • 测试更早介入，降低后期修复成本
  • 每个开发阶段都有对应的测试阶段
  • 强调验证和确认

迭代与增量

迭代和增量是现代软件开发的重要概念，它们改变了传统的线性开发模式。

迭代开发

迭代开发是将软件开发过程划分为多个循环迭代的过程。每个迭代都是一个完整的开发周期，包括需求分析、设计、实现、测试等活动。通过迭代，可以逐步完善系统功能。

增量开发

增量开发是将系统功能分解为多个增量模块，每次增量交付一个可运行的部分功能。增量模型允许先交付核心功能，再逐步增加其他功能。

迭代与增量的结合

现代敏捷方法通常结合使用迭代和增量，如Scrum框架中的Sprint就是典型的迭代增量开发模式。

螺旋模型

螺旋模型是一种风险驱动的迭代过程模型，它将风险分析与软件开发过程相结合。

螺旋模型的特点

  • 迭代性：多次循环逐步完善系统
  • 风险驱动：每次迭代都进行风险分析
  • 灵活性：可以根据项目实际情况调整
  • 用户参与：强调用户参与每个阶段

螺旋模型的优势

  • 早期识别和降低风险
  • 灵活性高，适应需求变化
  • 用户满意度高
  • 降低项目失败风险

螺旋模型的适用场景

  • 大型复杂系统
  • 高风险项目
  • 需求不确定的项目

构件组装模型

构件组装模型是一种基于可复用构件的软件开发方法，通过组装已有的软件构件来构建新系统。

构件组装模型的优点

  • 易扩展：通过添加新构件扩展系统功能
  • 易重用：构件可在多个项目中复用
  • 降低成本：减少重复开发
  • 任务灵活：开发任务安排更灵活

构件组装模型的缺点

  • 构件设计需要经验丰富的架构师
  • 设计不好的构件难以重用
  • 过度强调重用可能牺牲性能
  • 第三方构件质量难以控制

典型示例

  • 方舱医院：标准化构件快速组装
  • 乐高积木：标准构件无限组合

基于构件的软件工程

基于构件的软件工程（CBSE）

CBSE体现了”购买而不是重新构造”的哲学，强调使用现成的商业构件或开源构件来构建软件系统。

CBSE的构件应具备的特征

  • 可组装性：所有外部交互必须通过公开定义的接口进行
  • 可部署性：构件是二进制形式，能作为独立实体在平台上运行
  • 文档化：用户根据文档判断构件是否满足需求
  • 独立性：可以在无其他特殊构件的情况下进行组装和部署
  • 标准化：符合某种标准化的构件模型

构件的组装方式

  1. 顺序组装：按顺序调用已存在的构件，可以用两个已存在的构件创造新构件
  2. 层次组装：被调用构件的”提供”接口必须和调用构件的”请求”接口兼容
  3. 叠加组装：多个构件合并形成新构件，新构件整合原构件功能，对外提供新接口

快速应用开发模型

快速应用开发（RAD）模型是一种强调极短开发周期的软件过程模型，结合了瀑布模型和构件组装模型的优点。

RAD模型的特点

  • 强调快速迭代和用户反馈
  • 基于构件的快速组装
  • 时间box驱动，限制开发时间
  • 强调用户参与

RAD模型的适用场景

  • 需求相对稳定
  • 业务逻辑清晰
  • 需要快速交付
  • 中小型项目

统一过程

统一过程（UP）是一种用例驱动的迭代增量软件过程框架，由Rational软件公司创建，后来被IBM收购。

统一过程的特点

  • 用例驱动：以用例为核心组织需求和设计
  • 以架构为中心：强调早期构建稳定架构
  • 迭代增量：通过多次迭代交付增量
  • 风险驱动：优先处理高风险部分

统一过程的阶段

  1. ** inception（初始阶段）**：确定项目范围和目标
  2. elaboration（细化阶段）：分析需求，构建架构
  3. construction（构建阶段）：详细设计和编码
  4. transition（交付阶段）：测试和部署

统一过程的工件

统一过程定义了多种工件，包括用例模型、分析模型、设计模型、实现模型、测试模型等。

敏捷方法概述

敏捷方法是一类以人为核心、迭代增量的软件开发方法，强调快速响应变化和持续交付价值。

敏捷方法的核心原则

  • 个体和互动高于流程和工具
  • 可工作的软件高于详尽的文档
  • 客户合作高于合同谈判
  • 响应变化高于遵循计划

敏捷方法

极限编程（XP）

极限编程是最具代表性的敏捷方法，采用近螺旋式的开发模式。

四大价值观

  • 沟通：加强面对面沟通，减少书面文档
  • 简单：不过度设计，追求简单实现
  • 反馈：及时获取反馈，快速调整
  • 勇气：接受变更的勇气，敢于重构

十二条过程实践规则

  • 简单设计、测试驱动、代码重构
  • 结对编程、持续集成、现场客户
  • 发行版本小型化、系统隐喻
  • 代码集体所有制、规划策略
  • 规范代码、40小时工作机制

其他敏捷方法

  • 水晶方法：提倡”机动性”的方法，拥有对不同类型项目有效的敏捷过程
  • SCRUM：侧重于项目管理的敏捷框架
  • 特征驱动开发方法（FDD）：认为有效软件开发需要三要素【人、过程、技术】
  • 开放式源码：程序开发人员在地域上分布很广
  • ASD方法：核心是三个非线性的、重叠的开发阶段：猜测、合作与学习
  • 动态系统开发方法（DSDM）：倡导以业务为核心

逆向工程

逆向工程是分析已有程序，力图在比源代码更高抽象层次上建立程序表示的过程。

与逆向工程相关的概念

  1. 重构/重组：在同一抽象级别上转换系统描述形式
  2. 设计恢复：借助工具从已有程序中抽象出数据设计、总体结构设计和过程设计等信息
  3. 逆向工程：分析程序，在更高抽象层次上建立表示，是设计恢复的过程
  4. 正向工程：从现有系统恢复设计信息，并使用该信息重构或改善系统
  5. 再工程：对现有系统的重新开发过程，包括逆向工程、新需求考虑和正向工程三个步骤

净室软件工程

净室软件工程是一种强调在开发初期就确保软件质量的软件工程方法。

净室软件工程的核心思想

  1. 净室即无尘室、洁净室，是一个受控污染级别的环境
  2. 使用盒结构规约（或形式化方法）进行分析和设计建模
  3. 强调正确性验证是发现和消除错误的主要机制
  4. 使用统计测试获取软件可靠性认证所需的出错率信息

净室软件工程技术手段

  1. 统计过程控制下的增量式开发：控制迭代
  2. 基于函数的规范和设计：盒子结构
    • 行为视图（黑盒）
    • 有限状态机视图（状态盒）
    • 过程视图（明盒）
  3. 正确性验证：净室工程的核心
  4. 统计测试和软件认证：使用统计学原理，总体太大时采用抽样方法

需求工程

需求工程概述

需求工程是软件工程的重要阶段，包括需求获取、需求分析、需求规格说明、需求验证和需求管理等活动。

需求分类

软件需求通常分为：

  • 功能需求：系统应该做什么
  • 性能需求：系统的响应时间、吞吐量等
  • 接口需求：与其他系统的交互
  • 数据需求：数据的格式、存储等
  • 安全需求：访问控制、加密等
  • 可靠性需求：容错能力、可用性等

需求获取方法

常用的需求获取方法包括：

  • 访谈、问卷调查
  • 观察和文档分析
  • 原型法
  • 头脑风暴
  • 用例分析

结构化需求分析

结构化需求分析使用数据流图（DFD）、实体关系图（ERD）等工具分析系统需求。

UML基本概念

统一建模语言（UML）是面向对象建模的国际标准，它定义了多种图形符号来描述软件系统的不同方面。

UML的4+1视图

UML的4+1视图从不同角度描述系统：

  • 用例视图：描述系统功能需求
  • 逻辑视图：描述系统的静态结构和动态行为
  • 实现视图：描述系统的代码组织和构件
  • 部署视图：描述系统的物理部署
  • 进程视图：描述系统的并发和同步

需求定义

严格定义法

严格定义法基于以下假设：

  • 所有需求都能够被预先定义
  • 开发人员与用户之间能够准确而清晰地交流
  • 采用图形/文字可以充分体现最终系统

原型法

原型法基于以下假设：

  • 并非所有需求都能在开发前被准确说明
  • 项目参加者之间通常存在交流困难
  • 需要实际的、可供用户参与的系统模型
  • 有合适的系统开发环境
  • 反复是完全需要和值得提倡的

需求验证

需求验证是确保需求规格说明书正确性、完整性和一致性的活动。

需求验证的方法

  • 需求评审
  • 原型验证
  • 测试用例生成
  • 需求跟踪

需求验证的目标

  • 需求完整
  • 需求一致
  • 需求可行
  • 需求可测试

需求跟踪

需求跟踪是建立和维护需求与其他工件之间关系的过程。

需求跟踪的类型

  • 前向跟踪：从需求到设计和代码
  • 后向跟踪：从设计代码回溯到需求
  • 双向跟踪：前后双向跟踪

需求变更管理

需求变更管理是控制需求变更的过程，确保变更得到合理评估和控制。

需求变更管理的流程

  1. 变更申请
  2. 变更评估
  3. 变更审批
  4. 变更实施
  5. 变更验证

软件系统建模

1. 结构化建模方法

结构化建模方法是以过程为中心的技术，可用于分析现有系统及定义新系统的业务需求。结构化建模方法所绘制的模型称为数据流图（DFD），适用于流程较为稳定的系统。

2. 信息工程建模方法

信息工程建模方法是一种以数据为中心但过程敏感的技术，强调在分析过程需求之前首先研究和分析数据需求。信息工程建模方法所创建的模型被称为实体联系图（ERD），主要用于数据建模。

3. 面向对象建模方法

面向对象建模方法将”数据”和”过程”集成到被称为”对象”的结构中，消除了数据和过程的人为分离。面向对象建模方法所创建的模型被称为对象模型。UML（统一建模语言）是目前最常用的面向对象建模标准。

软件界面设计

人机界面设计的黄金三法则

人机界面设计应遵循以下三项核心原则：

1. 置于用户控制之下

  • 以不强迫用户进入不必要或不希望动作的方式定义交互方式
  • 提供灵活的交互方式
  • 允许用户交互被中断和撤销
  • 当技能级别增加时使交互流水化并允许定制交互
  • 使用户隔离内部技术细节
  • 设计应允许用户和出现在屏幕上的对象直接交互

2. 减少用户的记忆负担

  • 减少对短期记忆的要求
  • 建立有意义的缺省
  • 定义直觉性的快捷键
  • 界面的视觉布局应该基于真实世界的隐喻
  • 以不断进展的方式揭示信息

3. 保持界面的一致性

  • 允许用户将当前任务放入有意义的语境
  • 在应用系列内保持一致性
  • 如过去的交互模型已建立用户期望，除非有迫不得已的理由，不要改变它

结构化设计

概要设计与详细设计

  • 概要设计（外部设计）：将功能需求分配给软件模块，确定每个模块的功能和调用关系，形成模块结构图
  • 详细设计（内部设计）：为每个具体任务选择适当的技术手段和处理方法

结构化设计原则

  • 模块独立性原则：高内聚、低耦合
  • 保持模块大小适中
  • 多扇入、少扇出
  • 深度和宽度均不宜过高

模块的四个要素

  1. 输入和输出：模块的输入来源和输出去向都是同一个调用者
  2. 处理功能：模块把输入转换成输出所做的工作
  3. 内部数据：仅供该模块本身引用的数据
  4. 程序代码：用来实现模块功能的程序

面向对象设计

面向对象设计的基本过程

面向对象设计将面向对象分析的结果转化为软件结构，包括类设计、对象交互设计、关系设计等活动。

面向对象设计原则（SOLID）

  • 单一职责原则：设计目的单一的类
  • 开放-封闭原则：对扩展开放，对修改封闭
  • 里氏替换原则：子类可以替换父类
  • 依赖倒置原则：依赖于抽象，而不是具体实现
  • 接口隔离原则：使用多个专门的接口比使用单一总接口好
  • 组合重用原则：尽量使用组合而不是继承关系达到重用目的
  • 迪米特原则（最少知识原则）：一个对象应当对其他对象有尽可能少的了解

总结

软件工程是一门研究如何系统化、规范化、可量化开发和维护软件的学科。本文介绍了软件工程的核心概念、主要过程模型、设计原则等内容。

软件过程模型是软件开发的方法论骨架，从传统的瀑布模型到现代的敏捷方法，每种模型都有其适用场景。瀑布模型适合需求明确的项目，螺旋模型适合高风险项目，敏捷方法则适合需求变化频繁的项目。

需求工程是软件开发的起点，需求的质量直接决定了软件产品的质量。结构化设计和面向对象设计是两种主要的软件设计方法，其中面向对象设计原则（SOLID）为高质量软件设计提供了指导。

人机界面设计影响着用户体验，遵循”黄金三法则”可以帮助设计出用户友好的界面。随着技术的发展，软件工程的方法和工具也在不断演进，开发者需要持续学习和实践。
