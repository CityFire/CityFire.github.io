性能优化原理

  • CPU: 计算视图frame，图片解码，需要绘制纹理图片通过数据总线交给GPU
  • GPU: 纹理混合，顶点变换与计算,像素点的填充计算，渲染到帧缓冲区。
  • 时钟信号：垂直同步信号V-Sync / 水平同步信号H-Sync。
  • iOS设备双缓冲机制：显示系统通常会引入两个帧缓冲区，双缓冲机制

启动

在展开来说之前，我们需要定义下什么是App的启动：
通俗来讲，就是从用户点击App图标开始，到看到第一个页面的时间间隔。
细分来看，App的启动分两种：冷启动和热启动。

  • 冷启动：App启动前，内核没有为它分配相应的进程。
  • 热启动：App冷启动后，用户将App退出至后台，再进入的过程。
    热启动没什么好说的，能做的通用的事情非常少。App从后台回来直接进入AppDelegate生命周期函数，剩下的就是App自身的业务逻辑，如果要优化，也仅是业务上的优化，恢复数据，做一些同步。所以本篇我们只展开讲冷启动(后文将用启动代替冷启动)的流程和优化。BTW: 笔者建议先从热启动开始优化，毕竟大部分的性能瓶颈都在自家代码里。

App启动步骤概览
从大方向讲，App启动步骤分为三个阶段：

  1. 从系统exec()调用到我们App的main()为止 (main函数之前)
  2. main()执行之后
  3. 首屏渲染结束(至第一个界面的viewDidAppear:，nib loading)
    整个流程如下：

概括来讲：内核先将我们的App加载进内存，之后加载一个“中间件”: dynamic loader(简称:dyld)。之后dyld会负责分析App的Mach-O文件以加载所需的dynamic libraries。之后利用Rebasing和Binding修正内部和外部指针指向(ASLR机制)。最后加载runtime组件，runtime组件加载好后就会向需要初始化的object发送消息，开始初始化。 至此，main函数之前的步骤结束。之后的流程就是我们耳熟能详的生命周期了，App开始在AppDelegate里面初始化我们自定义的服务以及渲染首屏等。

简单来说iOS启动分为加载Mach-O和运行时初始化过程，加载Mach-O会先判断加载的文件是不是Mach-O，通过文件第一个字节，也叫魔数来判断，当是下面四种时可以判定是Mach-O文件：

  • 0xfeedface对应的loader.h里的宏是MH_MAGIC
  • 0xfeedfact宏是MH_MAGIC_64
  • NXSwapInt(MH_MAGIC)宏MH_GIGAM
  • NXSwapInt(MH_MAGIC_64)宏MH_GIGAM_64

Mach-O主要分为：

  • 中间对象文件（MH_OBJECT）
  • 可执行二进制（MH_EXECUTE）
  • VM 共享库文件（MH_FVMLIB）
  • Crash 产生的Core文件（MH_CORE）
  • preload（MH_PRELOAD）
  • 动态共享库（MH_DYLIB）
  • 动态链接器（MH_DYLINKER）
  • 静态链接文件（MH_DYLIB_STUB）符号文件和调试信息（MH_DSYM）这几种。

确定是Mach-O后，内核会fork一个进程，execve开始加载。检查Mach-O Header。随后加载dyld和程序到Load Command地址空间。通过 dyld_stub_binder开始执行dyld，dyld会进行rebase、binding、lazy binding、导出符号，也可以通过DYLD_INSERT_LIBRARIES进行hook。

dyld_stub_binder给偏移量到dyld解释特殊字节码Segment中，也就是真实地址，把真实地址写入到la_symbol_ptr里，跳转时通过stub的jump指令跳转到真实地址。dyld加载所有依赖库，将动态库导出的trie结构符号执行符号绑定，也就是non lazybinding，绑定解析其他模块功能和数据引用过程，就是导入符号。

Mach-O

Mach-O，全称是Mac object file format, 是一种文件类型。哪些文件是Mach-O呢？

  • Exectuable: 例如我们App bundle下的二进制文件
  • Dylib: 动态库，好比Windows下的DLL
  • Bundle: 指的是无法被链接，只能使用dlopen加载的动态库，例如Mac平台下的plug-ins
  • Image: 以上三个
  • Framework: 包含资源文件、头文件等的dylib

在 WWDC 2017 上苹果重申了几个在App启动时需要优化的点：

  • 尽量减少嵌入的dylibs的数量
  • 推荐使用系统提供的库
  • 尽量减少类和方法的数量
  • 尽量减少初始化函数
  • 尽量使用Swift
  • Swift没有初始化器。
  • Swift不允许特定类型的未对齐数据结构（这样的结构会延长启动时间。）
  • Swift代码更精简，所以性能更好

优化卡顿

图片

1、减少离屏幕渲染操作，layer.shadow、layer.allowsGroupOpacity or layer.allowsEdgeAntialiasing、layer.shouldRasterize、layer.mask、layer.masksToBounds && layer.cornerRadius
阴影：绘制shadow，使用ShadowPath来替代shadowOffset等属性的设置
圆角：利用GraphicsContex生成一张带圆角的图片；使用一张中间透明图片蒙上去；有时候只需要设置layer.cornerRadius即可。
2、像素对齐：图片位置以及大小不要出现小数点。
3、图片格式：打包进程序的图尽量为32bit/通道的图片。苹果的GPU只解析32bit的颜色格式。其他格式，要先经过CPU处理，才能解析。
4、避免缩放：UIImageView的frame大小跟图片原始尺寸不一致导致拉伸，导致消耗资源压缩或者放大图片
5、后台读取：图片读取操作需要放在异步线程
6、后台解码：iOS 读取png图片后，是不能直接显示的，需要先解码。解码后得到的位图才能显示。如果不主动异步解码，那么系统会主动在主线程执行解码操作。
7、图片缓存：缓存适当数量的图片，从内存中获取，增加访问速度。

数据库

1、将频繁读写数据库操作改成缓存，比如声音下载进度更新，点击进入直播间频繁读取数据库
2、将数据库读写操作放入子线程
总结：减少不必要的数据库操作 + 必要操作放入子线程

UITableView

1、size缓存：包括cell高度，cell中各个控件的宽高等
2、cell中图片，是否遵循了图片优化中7条规则
3、cell中的数据，是否在异步处理，如读取数据库。
4、滑动时按需加载，这个在大量图片展示，网络加载的时候很管用
5、异步绘制，遇到复杂界面，遇到性能瓶颈时，可能就是突破口，参考动态列表；

总结

FPS优化的核心是：把一切可以在异步线程执行的操作，从主线程中移除。
I/O操作放入异步线程处理。如：沙盒文件读写，数据库读写等。
图片在后台下载，后台写磁盘，后台解压，后台处理圆角等。
缓存图片和一些复杂耗时的运算结果。
延时加载，按需加载。如：多页内容用UICollectionView,不用UIScrollView。

二、编译时间优化

1 一个头文件中包含多个头文件的情况，要谨慎使用，最好使用粒度更细的头文件

2 PCH中的文件，会被所有文件依赖，修改后，会彻底重新编译。增加编译成本

3 debug模式下，关闭dSYM选项。可节省30s的时间，并且可以去除200个警告，不影响Xcode调试，但会影响Profile工具

编译后得到的可执行文件中不会包含调试符号，只会包含.o文件的引用。
调试符在.o文件中。
编译结束时，会调用dsymutil 将.o中的符号抽出来，生成.dSYM文件
第三方库AgoraRTCEngine ，编译的时候开启了Generate Debug Symbols
导致二进制文件中包含了.o引用。
我们在执行dsymutil命令生成dSYM时，找不到这些.o文件，产生警告。

三、具体优化点

一个LizhiFM.pb.h头文件中包含了27个细分的.pb.h头文件，LizhiFM.pb.h 在工程中import次数，高达512次。 导致 512 * 27 次的文件内容处理。

1 处理方案，采用更细粒度的文件引入

1 对已经引入LizhiFM.pb.h文件，直接删除引用。编译会报错，根据提示，用具体的头文件代替。如 import “Ad416.pb.h“。

2 新增的网络Scene处理，ITFeedBackScene为例，关键字为FeedBack。根据规则合成Response+关键字 = ResponseFeedBack。 按住shift+command+o,输入ResponseFeedBack，就可以查找到它定义在UserCommon43.pb.h中。直接引用它。

2 效果

工程中的#import “LizhiFM.pb.h”全部被删除，改用了具体业务头文件.ph.h代替后，编译时间缩短了11分钟

develop 耗时15.65min
优化后，节省11分钟的时间，编译速度加快了70%
